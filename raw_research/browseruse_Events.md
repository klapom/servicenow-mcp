Based on the ServiceNow documentation, here is the comprehensive information regarding Events and Event Management:

### 1. System Events & gs.eventQueue
*   **Definition**: System events are special records in the `sysevent` table used to log occurrences and trigger automated responses (notifications, script actions, workflows).
*   **gs.eventQueue()**: This method is used in server-side scripts (usually Business Rules) to insert an event into the queue.
    *   **Syntax**: `gs.eventQueue('event_name', GlideRecord, parm1, parm2, queue_name);`
*   **Event Registration**: New events must be registered in the **Event Registry** (`sysevent_register` table). This allows the system (and UI pickers for Notifications/Script Actions) to recognize and respond to the event.

### 2. Event Parameters & sysevent Table
*   **sysevent Table**: Often called the Event Log or Event Queue. It stores the event records until they are processed.
*   **Fields**:
    *   **Name**: The unique identifier for the event (e.g., `incident.commented`).
    *   **Parm1 & Parm2**: Optional parameters used to pass contextual data (like a User Sys ID or a specific value) to the handler.
    *   **Table**: The table the event is associated with.
    *   **Instance**: The Sys ID of the specific record that triggered the event.
    *   **State**: Events can be in states like **Ready** (waiting), **Processed** (completed), **Error** (failed), or **Transferred**.

### 3. Script Actions
*   **Definition**: Server-side scripts triggered exclusively by events.
*   **Objects available**:
    *   `event`: The GlideRecord of the event itself (access via `event.parm1`, `event.parm2`).
    *   `current`: The GlideRecord of the record that triggered the event.
*   **Usage**: Used for complex logic that shouldn't run immediately in a Business Rule, such as creating related records or performing external integrations after a record change.

### 4. Events vs. Business Rules
*   **Business Rules**: Run immediately (before/after) database operations. Use them for synchronous logic that affects the current transaction.
*   **Events**: Use them for **asynchronous** processing. If a task (like sending an email or a complex calculation) doesn't need to happen instantly or might slow down the user's experience, fire an event and let a background process handle it.

### 5. Notification Events
*   Notifications can be configured to fire 'When an event is fired'.
*   This allows the notification to use `event.parm1` or `event.parm2` as recipients or to include their values in the email body using `${event.parm1}`.

### 6. Custom Events & Best Practices
*   **Creation**: 1. Register the event in the Event Registry. 2. Create a trigger (usually a Business Rule using `gs.eventQueue`). 3. Create a handler (Notification or Script Action).
*   **Best Practices**:
    *   Always use a clear naming convention (e.g., `table_name.description`).
    *   Avoid firing too many events in a loop to prevent queue congestion.
    *   Use `parm1` and `parm2` for dynamic data rather than hardcoding values.

### 7. Event Log & Debugging
*   **Event Log**: Navigate to `System Policy > Events > Event Log` to see all fired events.
*   **Debugging**: Check the **State** field for 'Error'. Review the **Processing Time** to identify performance bottlenecks. Use `gs.log()` within Script Actions to trace execution.

### 8. Scheduled Events
*   Events can be generated by **Scheduled Script Executions** (Scheduled Jobs) to trigger periodic tasks, such as sending reminders for overdue tasks by querying the database and calling `gs.eventQueue` for each matching record.
