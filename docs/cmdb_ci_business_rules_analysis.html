Erkannte Probleme & Widersprueche

Severity Rule Problem Auswirkung auf FNT
MEDIUM Update location as needed
Order 100, before INSERT+UPDATE		 Ueberschreibt location unkontrolliert mit assigned_to.location bei JEDEM Insert/Update — auch wenn Location absichtlich anders gesetzt wurde. Kein direkter Konflikt — unsere FNT-Felder (u_fnt_campus, u_fnt_gebaeude) sind separat. Aber wenn spaeter Location-Mapping kommt (OP-5), muss diese Rule beachtet werden.
HIGH Populate Related Countries
Order 100, before INSERT+UPDATE		 countryArrayIDs ist als Array [] initialisiert, wird mit += String-konkateniert. Erzeugt trailing Komma. JavaScript Typ-Coercion maskiert den Bug. Kein direkter FNT-Bezug, aber feuert bei jedem CI-Update (auch FNT-initiiert).
MEDIUM Lifecycle Sync (2 Rules)
Order 9000, before INSERT+UPDATE		 Bidirektionaler Sync zwischen Legacy und CSDM Lifecycle. Endlosschleifen-Risiko, nur durch interne LifeCycleUtil Logik verhindert. Direkt relevant: FNT Retire (Section 2.3) setzt life_cycle_stage -> triggert beide Rules. FNT Timestamp Check MUSS vor Order 9000 laufen (Order 50 ist korrekt).
LOW GxP Relevance (cmdb_ci_appl)
Order 100, before INSERT+UPDATE		 Enthaelt gs.log("TEST MS 1:") Debug-Statements. Traversiert Relations ineffizient (nested GlideRecord queries). Kein FNT-Bezug, aber Debug-Logs sollten entfernt werden.
LOW Leere Rules (2 Stueck)
cmdb_ci_business_app		 "Reset Platform Host" und "Reset stale retired status" haben leere Script-Bodies (// Add your code here). Toter Code. Kein Effekt, aber irrefuehrend.

FNT Rules Platzierung — Order Begründung

FNT Rule When Order Begruendung
FNT Idempotency Check before INSERT 10 Muss als ERSTES laufen — vor jeder anderen Logik (Asset-Erstellung, Kategorie etc.). Bricht ab wenn Duplikat.
FNT Timestamp Check (LWW) before UPDATE 50 Muss VOR Location-Update (100), Manufacturer-Sync (742) und Lifecycle-Sync (9000) laufen. Verhindert stale Updates bevor andere Rules Seiteneffekte erzeugen.
FNT Outbound Sync after UPDATE 200 NACH dem relationship.rollup Event (100) und NACH Track Retired (100), aber VOR Model Category (500) und Duplikat-Move (1000). Stellt sicher dass alle OOTB before-Rules gelaufen sind bevor der Delta-Payload berechnet wird.
